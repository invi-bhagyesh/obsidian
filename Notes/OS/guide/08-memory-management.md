# 08 — Memory Management

## 8.1 Background

A process executing in memory consists of:
- **Executable instructions**, **stack**, **heap**, plus state in OS kernel (registers, open files, related processes).

Multiple processes share RAM; each has its own memory map.

### Key facts
- Program must be brought from disk into memory and placed within a process for it to be run.
- Main memory and registers are the only storage CPU can access **directly**.
- Memory unit only sees a stream of addresses + read or address+data + write.
- Register access: 1 CPU clock; main memory: many cycles → **stall**.
- **Cache** sits between memory and registers.
- **Memory protection** required for correct operation.

### Base & Limit Registers
A pair of **base** and **limit registers** define the logical address space. Every memory access in user mode must satisfy:
$$\text{base} \le \text{address} < \text{base} + \text{limit}$$

**Numerical:** Layout: OS at 0–256000, processes filling memory; one process at base = 300040, limit = 120900 → its valid range is 300040 to 420940.

### Hardware address protection
```
CPU address → (≥ base?) → (< base+limit?) → memory
              ↓ no       ↓ no
          trap to OS   trap to OS
```

## 8.2 Address Binding & Multistep Compilation

```
source program  → compiler/assembler   [compile time]
   → object module + others → linkage editor
   → load module → loader + system library  [load time]
   → in-memory binary image (with dynamic linking)  [execution time]
```

Addresses are bound at one of three stages:
- **Compile time** — generates absolute code; must recompile if start address changes.
- **Load time** — relocatable code; bound when program is loaded.
- **Execution time** — binding deferred to runtime if process can be moved between memory locations during execution; requires hardware support (MMU).

Addresses representations:
- Source code → symbolic.
- Compiled → relocatable (e.g., "14 bytes from beginning of this module").
- Linker/loader → absolute (e.g., 74014).

---

## 8.3 Logical vs Physical Address Space

- **Logical (virtual) address** — generated by CPU.
- **Physical address** — seen by the memory unit.
- Same in compile- and load-time binding; **different** at execution-time binding.
- **Logical address space** — set of all logical addresses.
- **Physical address space** — set of all physical addresses.

---

## 8.4 Dynamic Loading & Linking

### Dynamic Loading
- A routine is not loaded until called.
- Better memory utilization; unused routines never loaded.
- All routines kept on disk in relocatable load format.
- No special OS support required (just program design).

### Dynamic Linking
- **Static linking** — libraries combined with program at link time.
- **Dynamic linking** — postponed until execution.
  - A small **stub** locates the library routine.
  - Stub replaces itself with the routine's address and executes the routine.
  - If routine not yet in process address space, OS adds it.
- Useful for libraries; system also called **shared libraries**.

---

## 8.5 Memory-Management Unit (MMU)

Hardware that maps virtual to physical addresses at run time.
- Simple scheme: a **relocation register** holds an offset added to every logical address.
- The user program deals with logical addresses; never sees physical.

### Numerical — relocation register
Relocation = 14000; logical address = 346 → physical = 14000 + 346 = **14346**.

---

## 8.6 Swapping

A process can be swapped temporarily out to a **backing store** (fast disk) and brought back later.
- Lets total memory of all processes exceed physical memory.
- **Roll-out, roll-in** — used in priority scheduling: swap out a low-priority process to load a high-priority one.
- Major time = transfer time, proportional to memory swapped.

### Numerical — context switch with swapping
- Process: 100 MB; transfer rate: 50 MB/sec.
- Swap out: 100/50 = **2000 ms**.
- Plus swap in: 2000 ms.
- **Total context switch component = 4000 ms (4 sec)**.

Reductions: track how much memory is actually used (system calls `request_memory()`/`release_memory()`).

---

## 8.7 Contiguous Memory Allocation

Main memory split into two partitions:
- Resident OS (low memory + interrupt vector).
- User processes (high memory; each process in single contiguous section).

Relocation registers protect processes from each other and from the OS.

### Multiple-partition allocation
- Variable-partition sizes.
- **Hole** — block of available memory; holes scattered throughout.
- Process arriving allocated a hole large enough; process exiting frees its partition; **adjacent free partitions combined**.

### Dynamic Storage Allocation Strategies
- **First-fit:** Allocate the first hole that is big enough.
- **Best-fit:** Allocate the smallest hole that is big enough → smallest leftover.
- **Worst-fit:** Allocate the largest hole → largest leftover.

### Fragmentation
- **External fragmentation:** total free memory exists but is non-contiguous.
- **Internal fragmentation:** allocated memory slightly larger than requested.

**Numerical:**
| Address | Size | Status |
|---------|------|--------|
| 0       | 120k | P4 |
| 120k    | 60k  | free |
| 180k    | 20k  | P5 |
| 200k    | 10k  | free |

Total free = 70k; new process P6 needs **65k** but cannot fit (largest contiguous free block is 60k) → external fragmentation.

### Compaction
Reduce external fragmentation by shuffling memory contents → all free memory in one block. **Possible only if relocation is dynamic at execution time.**

## 8.8 Segmentation

Memory-management scheme that supports **user view** of memory: a program is a collection of segments (main, procedure, function, object, stack, symbol table, arrays, ...).

### Common segments
**Text, Data, Heap, Stack** — typical user-space layout.

### Segmentation Architecture
- Logical address = ⟨segment-number, offset⟩.
- **Segment table** — each entry has:
  - **base** — starting physical address of the segment.
  - **limit** — length of the segment.
- **STBR** (Segment-Table Base Register) — points to segment table.
- **STLR** (Segment-Table Length Register) — number of segments used; segment number $s$ valid iff $s < $ STLR.

### Translation
For logical (s, d):
- If d < limit[s]: physical = base[s] + d.
- Else: trap.

### Numerical (1)
Pointer to descriptor table = 0x3000; segment register = 1; offset = 100.

| Segment | Base | Limit |
|---------|------|-------|
| 0 | – | – |
| 1 | 1000 | 1000 |
| 2 | 4000 | 500 |
| 3 | 8000 | 1000 |
| 4 | 9000 | 1000 |

For segment 1, offset 100: 100 < 1000 → physical = 1000 + 100 = **1100**.

### Numerical (2) — trap detection

| Segment | Base | Length |
|---------|------|--------|
| 0 | 1219 | 700 |
| 1 | 2300 | 14  |
| 2 | 90   | 100 |
| 3 | 1327 | 580 |
| 4 | 1952 | 96  |

| Logical (s, d) | Check d < length? | Physical |
|----------------|-------------------|----------|
| (0, 430) | 430 < 700 ✓ | 1219 + 430 = **1649** |
| (1, 11)  | 11 < 14 ✓   | 2300 + 11 = **2311** |
| (2, 100) | 100 < 100 ✗ | **TRAP** |
| (3, 425) | 425 < 580 ✓ | 1327 + 425 = **1752** |
| (4, 95)  | 95 < 96 ✓   | 1952 + 95 = **2047** |

## 8.9 Paging

Physical address space of a process can be **noncontiguous**.

- Divide physical memory into fixed-sized blocks called **frames** (size = power of 2, typically 512 B – 16 MB; commonly **4 KB**).
- Divide logical memory into blocks of the same size called **pages**.
- Keep track of free frames.
- For an N-page program, find N free frames and load.
- A **page table** translates logical → physical addresses.
- Backing store also split into pages.
- Eliminates external fragmentation; **internal fragmentation** still possible (last page may not fully fill a frame).

### Address translation
Logical address = ⟨**page number p**, **page offset d**⟩.
- For logical address space $2^m$ and page size $2^n$: high $m-n$ bits = p, low $n$ bits = d.

```
logical: | p | d |
                ↓ page table lookup
physical: | f | d |
```

### Paging hardware
CPU generates (p, d) → page table[p] = f → physical address = (f, d).

### Paging example (n=2, m=4)
- 32-byte memory, 4-byte pages → 8 frames, 16-byte logical space (in this textbook example).
- Page table: 0→5, 1→6, 2→1, 3→2.

---

## 8.10 Paging — Numerical Examples

### Page table size (1)
> 32-bit logical address, 4 KB page size, 4-byte page-table entry.

- Logical space = $2^{32}$; pages = $2^{32}/2^{12} = 2^{20}$.
- Page table size = $2^{20} \times 4$ bytes = **4 MB**.

### Page table size (2)
> 64 MB physical memory, 32-bit virtual address, 4 KB page size.

- Virtual pages = $2^{20}$.
- Frames = $2^{26}/2^{12} = 2^{14}$ → frame number needs 14 bits ≈ 2 bytes/entry.
- Page table ≈ $2^{20} \times 2$ B = **2 MB**.

### Virtual to physical translation
> Page size = 2 KB. Page-frame mapping: 0→1, 1→3, 2→2, 3→0. Virtual address = 2500 (decimal).

- Page = 2500 / 2048 = 1; offset = 2500 − 2048 = 452.
- Page 1 → Frame 3. Physical = 3·2048 + 452 = 6144 + 452 = **6596**.

## 8.11 Page Table Implementation

- Page table in main memory.
- **PTBR** points to page table.
- **PTLR** indicates page table size.
- Naive scheme: every memory reference requires **two memory accesses** (page table + actual data).

### TLB (Translation Look-aside Buffer)
A fast-lookup hardware cache (associative memory) stores recently used (page → frame) mappings.

- Some TLBs store **address-space identifiers (ASIDs)** to avoid flushing on context switch.
- TLBs typically **64 to 1024 entries**.
- TLB miss → load value into TLB for next time.

### Effective Access Time (EAT)
Hit ratio = α; TLB lookup time = ε; memory access = m.

$$\text{EAT} = \alpha(\varepsilon + m) + (1 - \alpha)(\varepsilon + 2m)$$

### Numerical EAT (1)
α = 0.80, ε = 20 ns, m = 100 ns.
- Hit: 20 + 100 = 120; miss: 20 + 200 = 220.
- EAT = 0.80·120 + 0.20·220 = 96 + 44 = **140 ns**.

### Numerical EAT (2)
ε = 10 ms, m = 80 ms, α = 0.6.
- Hit: 90; miss: 170.
- EAT = 0.6·90 + 0.4·170 = 54 + 68 = **122 ms**.

### Numerical — solve for hit ratio
ε = 20 ns, m = 100 ns, EAT = 130 ns. Find α.

130 = α·120 + (1−α)·220
130 = 120α + 220 − 220α
−90 = −100α
**α = 0.9 (90%)**.

## 8.12 Memory Protection

- Protection bit per frame (read-only or read-write).
- **Valid–Invalid bit** in page table entry:
  - **valid** — page is in process's logical address space.
  - **invalid** — not in process's logical address space.
- Or use **PTLR** to bound page numbers.
- Violations → trap to kernel.

---

## 8.13 Shared Pages

### Shared (reentrant) code
- One read-only copy shared among processes (e.g., editors, compilers).
- Similar to threads sharing process space.
- Useful for IPC if read-write sharing allowed.

### Private code/data
- Each process has its own copy.
- Pages can appear anywhere in the logical address space.

---

## 8.14 Structure of Page Table — Avoiding Huge Tables

A 32-bit address with 4 KB pages and 4-byte entries → 4 MB per process for page table alone. Solutions:

### 1. Hierarchical (Multilevel) Paging
Logical address split: $|p_1|p_2|d|$ where $p_1$ indexes outer page table, $p_2$ indexes the inner page-of-page-table.

### 2. Hashed Page Tables
Hash the page number. The corresponding hash bucket holds a linked list of $(p, r, \ldots)$ entries; search list to find matching $p$.

### 3. Inverted Page Tables
One entry per **frame** rather than per page. Search the table for the (process, page) pair.

**Problem A.** A system uses paging with 1 KB pages. The page table is:
| page | frame |
|------|-------|
| 0 | 5 |
| 1 | 7 |
| 2 | 3 |
| 3 | 1 |

Translate virtual address 4900.
- Page = 4900/1024 = 4 → no page 4 → trap.

**Problem B.** EAT given TLB hit time 50 ns, mem time 200 ns, miss requires 2 mem accesses, hit ratio 90%.
- Hit: 50 + 200 = 250; miss: 50 + 400 = 450.
- EAT = 0.9·250 + 0.1·450 = 225 + 45 = **270 ns**.

**Problem C.** External fragmentation: free holes 100k, 500k, 200k, 300k, 600k. Process of 200k arrives.
- First-fit: places in 500k (first big enough), leftover 300k.
- Best-fit: places in 200k (exact), leftover 0.
- Worst-fit: places in 600k (largest), leftover 400k.
